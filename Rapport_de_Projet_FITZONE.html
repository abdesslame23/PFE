Résumé

Ce rapport de fin de formation décrit la conception et le développement de la plateforme web FITZONE, un système intégré de gestion administrative conçu pour moderniser les opérations des complexes sportifs et centres de fitness. L'application offre un portail interactif aux abonnés pour consulter les plannings de cours collectifs, s'inscrire en ligne et accéder à un guide d'entraînement par groupe musculaire.

La fonctionnalité phare du système est un mécanisme de pointage physique à l'entrée de la salle reposant sur un code personnel unique à 6 chiffres attribué à chaque abonné actif, validé en temps réel par l'API backend. Le système a été réalisé avec une architecture découplée associant un backend REST API sous le framework Laravel (PHP) et un frontend dynamique Single Page Application sous React (JavaScript), avec une base de données relationnelle MySQL.

Abstract

This end-of-training report presents the design and development of the FITZONE web platform, an integrated management system designed to modernize the administrative operations of fitness and sports centers. The application provides members with an interactive portal to view group class schedules, register online and access a muscle-group training guide.

The flagship feature is a physical entry check-in mechanism based on a unique 6-digit personal code assigned to each active subscriber, validated in real time by the backend API. The full system was built using a decoupled architecture combining a REST API backend powered by Laravel (PHP) and a SPA frontend using React (JavaScript), with a relational MySQL database.

Introduction Générale

L'essor considérable du secteur de la santé et du bien-être physique, conjugué à l'évolution rapide des technologies numériques, a profondément transformé les attentes des consommateurs vis-à-vis des services sportifs. Aujourd'hui, un abonné d'une salle de sport ne se contente plus d'un accès physique à des équipements : il recherche une expérience numérique fluide, intégrée et personnalisée qui prolonge sa relation avec l'établissement bien au-delà de ses séances d'entraînement.

Face à cet impératif de modernisation, les complexes sportifs qui maintiennent des processus manuels de gestion — registres papier, fiches d'abonnement, plannings affichés statiquement — se retrouvent en situation de désavantage compétitif croissant. Ils font face à des problèmes opérationnels récurrents : fraude aux abonnements par prêt de badges physiques, difficultés de suivi des paiements, impossibilité de réguler les inscriptions aux cours collectifs et manque de données analytiques exploitables.

C'est précisément dans ce contexte que s'inscrit le présent projet de fin de formation, réalisé au sein de l'Office de la Formation Professionnelle et de la Promotion du Travail (OFPPT), dont l'objet central est la conception et la réalisation de la plateforme FITZONE. Cette application web intégrée vise à centraliser l'ensemble des flux administratifs d'une salle de sport au sein d'un écosystème numérique unique, accessible depuis tout navigateur.

La problématique centrale de ce projet réside dans la sécurisation du contrôle d'accès physique. Mon approche innove en proposant un système de pointage par code personnel à 6 chiffres, généré à la création de chaque compte membre et validé en temps réel par une API sécurisée. Ce dispositif élimine les coûts liés aux badges physiques et rend impossible le prêt d'accès entre abonnés.

Ce rapport s'articule autour de quatre chapitres. Le Chapitre 1 pose le cadrage du projet et présente la planification Agile adoptée. Le Chapitre 2 analyse les solutions du marché et justifie mes choix technologiques. Le Chapitre 3 détaille ma modélisation UML. Enfin, le Chapitre 4 présente la réalisation concrète avec les interfaces réelles et les résultats des tests.

Chapitre 1
Cadrage du Projet

Ce chapitre pose les fondations conceptuelles du projet FITZONE, réalisé dans le cadre de ma formation en Développement Digital à l'OFPPT. Il couvre le contexte général, la problématique, les objectifs, les besoins fonctionnels et non fonctionnels, la méthodologie Agile/SCRUM adoptée ainsi que la planification à travers un backlog produit et un diagramme de Gantt.

1.1 Contexte et Problématique

La gestion opérationnelle d'une salle de sport moderne implique la coordination simultanée de plusieurs activités : enregistrement des abonnements, planification des cours collectifs, contrôle des accès à l'entrée et gestion des encaissements mensuels.

En l'absence d'un système d'information centralisé, ces tâches génèrent inévitablement des erreurs humaines coûteuses. La problématique du contrôle d'accès est particulièrement sensible : les solutions matérielles traditionnelles (badges magnétiques, bracelets RFID) impliquent un investissement initial important et présentent des failles de sécurité notoires — notamment le prêt ou la copie illicite des badges entre abonnés et personnes non inscrites.

Le manque de visibilité en temps réel sur les statistiques d'affluence et les revenus mensuels empêche également la direction de prendre des décisions opérationnelles éclairées. Ces constats justifient le développement d'une solution logicielle dédiée, économique et sécurisée.

1.2 Objectifs du Projet

L'objectif global de FITZONE est de fournir une plateforme web complète permettant d'automatiser et de sécuriser la gestion d'un centre sportif. Les objectifs spécifiques sont :

  • Portail membre self-service : Gestion du profil en autonomie, consultation de l'historique et réservation de cours.
  • Système de pointage innovant : Code personnel à 6 chiffres unique, validé instantanément par l'API Laravel.
  • Planning dynamique des cours : Vue interactive du calendrier avec gestion automatique des quotas de places.
  • Tableau de bord analytique : Indicateurs clés en temps réel pour l'administrateur (visites, revenus, abonnements).
  • Guide d'entraînement intégré : Catalogue d'exercices classés par groupe musculaire.

1.4 Méthodologie Adoptée (Agile / SCRUM)

Pour conduire ce projet, nous avons opté pour le cadre de développement Agile SCRUM. Notre projet a été découpé en trois sprints de deux semaines, chacun donnant lieu à une revue fonctionnelle avec l'encadrant.

  • Sprint 1 : Authentification JWT, génération du code de pointage, gestion des membres (CRUD).
  • Sprint 2 : Module de cours collectifs, réservation en ligne, validation du pointage à l'entrée.
  • Sprint 3 : Tableau de bord analytique, gestion des paiements, guide musculaire, tests et corrections.

2.2 Technologies et Frameworks Retenus

2.2.1 Frontend : React (JavaScript)

React est une bibliothèque JavaScript open-source maintenue par Meta. Fondée sur un système de composants UI réutilisables et un DOM virtuel, elle permet de concevoir des interfaces dynamiques Single Page Application (SPA) sans rechargement complet de page. Son large écosystème (React Router, Axios, Context API) et sa facilité de structuration en composants fonctionnels avec Hooks en font le choix idéal pour notre frontend interactif.

2.2.2 Backend : Laravel REST API (PHP)

Laravel est un framework PHP suivant l'architecture MVC. Il offre un routage API expressif, un ORM puissant (Eloquent) pour les interactions avec MySQL, un système d'authentification JWT et des outils de migration de schéma versionnés. Sa maturité et son excellente documentation en font un choix solide pour des API REST sécurisées.

2.2.3 Base de Données : MySQL

MySQL est un SGBD relationnel open-source. Sa robustesse, sa compatibilité native avec Laravel via Eloquent ORM et ses capacités de jointure entre tables (Utilisateurs, Abonnements, Visites, Cours) en font le choix optimal. Les index sur la colonne code_pointage garantissent des temps de réponse inférieurs à 50 ms pour la validation des accès.

Chapitre 4
Réalisation et Implémentation

Ce dernier chapitre présente les conditions de mise en place de l'environnement de développement, l'architecture logicielle, les détails d'implémentation clés, les captures d'écran réelles de l'application en fonctionnement, les résultats des tests et les difficultés techniques rencontrées.

4.1 Environnement de Développement

Le projet FITZONE fonctionne avec deux serveurs distincts démarrés simultanément :

  • Serveur Backend Laravel : Port 8000 via php artisan serve. Expose l'API REST JSON et communique avec MySQL.
  • Serveur Frontend React : Port 3000 via npm start. Les appels API sont centralisés dans un client Axios pointant vers http://localhost:8000/api.

La communication cross-origin est gérée par le middleware CORS configuré dans config/cors.php de Laravel.

4.2 Architecture Générale

L'application suit une architecture découplée client-serveur stricte. Le frontend React ne contient aucune logique métier : toutes les règles de validation, de sécurité et de persistance sont déléguées au backend Laravel. Cette séparation garantit une maintenabilité optimale et permet de faire évoluer les deux composants indépendamment.

4.3 Implémentation du Backend et du Frontend

Ci-dessous les extraits de code clés illustrant les algorithmes principaux de l'application FITZONE :

1. Validation du Pointage Physique (Laravel API)

// PointageController.php
public function valider(Request $request) {
    $user = User::where('code_pointage', $request->code)->with('abonnements')->firstOrFail();
    $abonnementActif = $user->abonnements->where('statut', 'paye')->where('date_fin', '>=', now())->first();

    if (!$abonnementActif) {
        return response()->json(['error' => 'Abonnement inactif'], 400);
    }
    Visite::create(['user_id' => $user->id, 'date_visite' => now()]);
    return response()->json(['nom' => $user->nom, 'status' => 'Accès autorisé']);
}

2. Authentification & Génération du Code à 6 Chiffres (Laravel JWT)

// AuthController.php
public function register(Request $request) {
    $codeUnique = str_pad(mt_rand(0, 999999), 6, '0', STR_PAD_LEFT);
    $user = User::create([
        'name' => $request->name, 'email' => $request->email,
        'password' => Hash::make($request->password), 'code_pointage' => $codeUnique
    ]);
    $token = auth()->login($user);
    return response()->json(['token' => $token, 'user' => $user], 201);
}

3. Intercepteur Axios Client avec Injection JWT (React Frontend)

// src/services/api.js
import axios from 'axios';
const api = axios.create({ baseURL: 'http://localhost:8000/api' });

api.interceptors.request.use((config) => {
    const token = localStorage.getItem('token');
    if (token) { config.headers.Authorization = `Bearer ${token}`; }
    return config;
});
export default api;

4.6 Difficultés Rencontrées et Solutions Apportées

Difficulté 1 — Anti-brute-force sur le code de pointage

Le clavier de saisie étant accessible sans authentification, nous avons configuré le Rate Limiting de Laravel, bloquant toute IP émettant plus de 5 requêtes par minute vers /api/pointage/valider.

Difficulté 2 — Race condition lors des réservations simultanées

Deux membres pouvaient réserver la dernière place d'un cours au même instant. Nous avons implémenté un verrou exclusif SQL (lockForUpdate() d'Eloquent) dans une transaction atomique pour éliminer ce risque.

Annexe B — Transaction Atomique pour la Réservation

Extrait de CoursController.php :

DB::transaction(function () use ($cours, $user) {
    $cours = Cours::where('id', $cours->id)->lockForUpdate()->first();

    if ($cours->inscrits >= $cours->places_max) {
        throw new Exception('Cours complet', 409);
    }

    $cours->increment('inscrits');
    Reservation::create(['user_id' => $user->id, 'cours_id' => $cours->id]);
});
